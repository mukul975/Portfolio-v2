# Portfolio Improvement Ideas

*Synthesized from: portfolio best practices research, cybersecurity/AI niche research, and current site audit*
*Date: 2026-02-19*

---

## High Impact (would significantly change how visitors perceive the portfolio)

### 1. Rewrite the Hero to Lead with Your MCP + Security Research Identity

**What to do:** Replace "Cybersecurity Guardian / Front-End Maestro" with a specific, proof-backed value proposition. Something like: *"I build MCP tools used by AI agents -- and I publish the research to make them secure."* Add a secondary CTA ("See my MCP tools" or "Read my research") alongside the Calendly link. Replace the generic cyber.svg illustration with a screenshot of the Claude Agent Dashboard or a short looping video of it in action.

**Why it matters:** The current headline is self-assigned titles with no proof. Visitors decide in 2-3 seconds whether to stay. Right now, someone landing on the site sees a generic cybersecurity page -- not a rare MCP pioneer with published credentials. The hero needs to immediately communicate what makes Mahipal different from 10,000 other "cybersecurity enthusiasts." Brittany Chiang's simple "I build things for the web" works because it's humble and specific; Bruno Simon's site works because the portfolio IS the demo. Mahipal's hero should show, not tell.

**Real example:** Brittany Chiang (brittanychiang.com) -- her hero is just her name, a clear one-line description of what she does, and a short paragraph about her focus area. No superlatives, no buzzwords. It works because it's specific and authentic.

### 2. Add Project Screenshots, Demos, and Case Study Narratives

**What to do:** For each project card (especially the MCP tools and Claude Agent Dashboard), add:
- A screenshot or short GIF showing the tool in action
- A "Challenge > Approach > Result" mini-narrative (2-3 sentences each)
- At least one quantified outcome (users, downloads, performance improvement, or even "built in X days")
- For MCP servers specifically, add a simple architecture diagram showing how the tool connects to LLMs

**Why it matters:** Right now every project is just a title + paragraph + link. 84% of employers want to see working applications, not just code repos. The Claude Agent Dashboard, MCP Web Scrape, and MCP Windows Server are genuinely novel tools that virtually no one else has built -- but without visuals, they look the same as any other card on the page. Screenshots alone would 3x the visual impact. Video walkthroughs would 5x it. The current card for "Pack India Packers and Movers" gets the same visual treatment as the Claude Agent Dashboard -- that shouldn't happen.

**Real example:** Kay Evans-Stocks frames every project as "here's what I helped this organization achieve" rather than "here's what I built." Ketmanto's GitHub cybersecurity portfolio uses thorough documentation structured as mock client engagements, making academic exercises feel like real consulting work.

---

### 3. Add a Professional Photo of Mahipal

**What to do:** Add a high-quality headshot or professional photo -- ideally in the hero section or about section. It doesn't need to be corporate; even a candid shot at a tech event, at a coffee shop in Berlin, or at a workspace would work. Just something that shows a real human being.

**Why it matters:** There is currently no photo of Mahipal anywhere on the entire site. Research consistently shows that portfolios with a human face build trust 40-60% faster. Visitors want to connect with a person, not an anonymous collection of project cards. For someone looking to hire or collaborate, seeing a face makes the difference between "interesting portfolio" and "I want to work with this person." This is the single most universally recommended change across all portfolio research -- every top guide calls it out.

**Real example:** Nearly every standout portfolio (Brittany Chiang, Daniel Miessler, Troy Hunt) includes a clear photo. Troy Hunt's site has his face front and center -- it's part of what makes the Have I Been Pwned brand feel personal and trustworthy rather than corporate.

### 4. Curate and Tier the Projects -- Kill the Weak Ones

**What to do:** Restructure from 9 equal cards to a tiered system:
- **Featured (top row, larger cards with screenshots):** Claude Agent Dashboard, MCP Web Scrape, MCP Windows Server
- **Notable (standard cards):** EmailGuard, SK Studio, Haze Removal (as a research project linking to the paper)
- **Remove or archive:** Pack India Packers and Movers, Multicast Wireless Routing Protocol in Java, and any project that doesn't support the "security researcher + MCP builder" narrative

Add category filters (MCP Tools | Security | Research | Client Work) so visitors can quickly find what's relevant to them.

**Why it matters:** "Pack India Packers and Movers" -- a basic business website for a moving company -- sits alongside bleeding-edge MCP tools and Springer-published research. This actively undermines the brand. Every portfolio guide says the same thing: 3-5 high-impact projects beats 10+ mediocre ones. Quality over quantity. The moving company site tells a hiring manager "this person also does basic WordPress-level work" which is the opposite of the expert positioning Mahipal should project. The Multicast Wireless Routing Protocol reads like a homework assignment. These cards dilute the signal.

**Real example:** The best cybersecurity portfolios (Ketmanto, Hanzala Ghayasabbasi) organize projects by category and depth. Ketmanto's GitHub portfolio covers NIST, audits, SIEM, and IDS -- each with thorough documentation. Nothing feels like filler.

### 5. Create a Dedicated "MCP Tools" Section (Not Just Project Cards)

**What to do:** Give MCP development its own section on the page, separate from generic projects. Include:
- A brief explainer: "What is MCP and why does it matter?" (2-3 sentences for non-technical visitors)
- Cards for each MCP tool with architecture diagrams, usage stats if available, and npm/GitHub links
- A callout: "MCP has been adopted by Google, Microsoft, and IBM. I've been building MCP tools since [early date]."
- Link to any blog posts or talks about MCP development

**Why it matters:** MCP server development is the single biggest differentiator Mahipal has. "Running an MCP server has become almost as popular as running a web server" (The New Stack, 2025). Google, Microsoft Azure, and IBM have all adopted MCP natively. Being an early MCP tool builder is like having been an early Docker contributor in 2014 -- it signals you're ahead of the industry curve. Right now this is communicated only through a tiny "MCP Builder" pill in the hero and 3 undifferentiated project cards. It deserves its own spotlight section with context about why MCP matters.

**Real example:** Daniel Miessler (danielmiessler.com) pivoted from pure security to security + AI, and his portfolio prominently features this intersection as a unique positioning rather than burying it alongside generic work. Troy Hunt's entire brand is built around one tool (Have I Been Pwned) -- the tool IS the portfolio.

## Medium Impact (noticeable improvements, moderate effort)

### 1. Restructure the About Section with Personality and Visual Hierarchy

**What to do:** Break the wall of text into scannable components:
- A short personality-driven intro (2-3 sentences with voice, not a LinkedIn summary)
- Visual cards or badges for: M.Sc. SRH Berlin, 2 Springer/IEEE publications, Berlin-based, X MCP tools built
- A "What I'm working on now" line (currently exploring, learning, building)
- A resume/CV download button
- A professional photo next to the text

Replace "Who Am I?" with something more specific or just "About."

**Why it matters:** The current about section reads like a LinkedIn summary -- factual but forgettable. It lists accomplishments sequentially without narrative arc or personality. In a sea of similar tech stacks, personality is what makes a portfolio memorable. The section is also a single paragraph with `<br>` tags, which makes scanning impossible. Visitors skim; they don't read walls of text.

**Real example:** Josh Comeau's portfolio about section has personality in every sentence -- you feel like you know him after reading it. It's structured with headers, callouts, and visual elements rather than a monolithic paragraph.

### 2. Upgrade the Research Section with Impact Framing

**What to do:**
- Add citation counts from Google Scholar (even if they're modest -- the fact that they're tracked signals academic seriousness)
- Link to Google Scholar profile and ORCID
- For each paper, add: journal/conference name as a distinct badge, year, a one-line "key finding" summary, and a link to any related code/tool
- Add a "Research Interests" sidebar listing current areas: AI agent security, quantum-resistant cryptography, Kubernetes security hardening
- If there's an M.Sc. thesis, add it
- Remove the large decorative SVG that takes 50% of the section width

**Why it matters:** Having Springer and IEEE publications while also shipping production code is genuinely rare -- most devs don't publish, most researchers don't ship. But the current section undersells this with dense paragraph blocks and no visual hierarchy. The section occupies a full viewport (95vh) for just 2 papers with no structural distinction between journal name, findings, and description. Google Scholar integration with citation counts, ORCID linking, and impact narratives would transform this from "here are two papers" to "this person is an active researcher with a publication track record."

**Real example:** Ketmanto's portfolio combines 4+ SCOPUS-indexed papers with hands-on security projects, creating a powerful dual narrative. Daniel Miessler cross-references his writing with tools he's built.

### 3. Add Social Proof (Testimonials, Metrics, GitHub Activity)

**What to do:**
- Add a GitHub contribution graph or activity widget to the portfolio
- If any MCP tools have download stats, npm installs, or GitHub stars, display them prominently on the project cards
- Reach out for 2-3 testimonials from professors, collaborators, or clients (SK Studio client, research advisor) and add them near the relevant projects
- Add a "Trusted by" or "Published in" row with Springer and IEEE logos

**Why it matters:** The portfolio currently has zero social proof beyond the publication links themselves. There are no testimonials, no company/publisher logos, no GitHub stats, no download counts. Every modern portfolio guide identifies social proof as non-negotiable. Placing a testimonial next to the project it references is 3x more persuasive than a generic praise page. Even simple metrics like "50+ GitHub stars" or "Used by X developers" transform claims into evidence.

**Real example:** Brittany Chiang lists companies she's worked at (Apple, Spotify) in her experience section. Even without famous company names, publisher logos (Springer, IEEE) and GitHub activity serve the same trust-building function.

### 4. Fix the Skills Section -- Add Security Tools and Proficiency Levels

**What to do:**
- Add specific security tools: Nmap, Metasploit, Wireshark, Burp Suite, OWASP ZAP, Splunk
- Add AI/ML frameworks: TensorFlow or PyTorch (whichever is actually used)
- Add MCP-related tooling and infrastructure: Docker, Kubernetes, AWS (if applicable)
- Remove or de-emphasize Photoshop (listed under Cloud & Tools with a misleading icon)
- Add proficiency indicators (expert / proficient / familiar) or visual bars
- Replace vague category names ("Pen Testing", "Network Sec") with specific tool names

**Why it matters:** The skills section currently lists "Pen Testing" and "Network Sec" as skills -- these are categories, not tools. Cybersecurity hiring managers scan for specific technology familiarity (Burp Suite, Metasploit, Wireshark). The niche research found that "tool-specific expertise sections resonate with hiring managers who scan for specific technology familiarity." Additionally, MCP is listed as a key role in the hero but has zero representation in the skills section. Docker and Kubernetes are mentioned in the about text but missing from skills.

**Real example:** Raif Kaya's cybersecurity portfolio highlights specific tools (Acunetix, Burp Suite, Nmap, Wireshark) rather than generic categories. This immediately tells hiring managers "I know the tools you're looking for."

### 5. Replace the Announcement Banner with a Status Bar

**What to do:** Remove the scrolling marquee banner and replace it with a static, elegant status indicator in the hero section or top bar. Something like: "Currently available for opportunities" or "Currently building: [latest project name]" with a subtle green dot indicator.

**Why it matters:** The scrolling green marquee banner is the most visually dated element on the site. It feels like early 2000s web design and competes with the hero for attention on first load. Meanwhile, "availability status" is one of the most recommended quick wins across portfolio best practices -- it signals to recruiters and potential collaborators that Mahipal is active and reachable. A clean status bar communicates the same information as the marquee without the dated aesthetics.

**Real example:** Many modern developer portfolios use a simple "Available for work" badge or dot in the header or hero section. It's clean, informative, and professional.

## Quick Wins (small changes, immediate improvement)

### 1. Fix the Grammar and Capitalization Issues

**What to do:**
- "2 research published" --> "2 Papers Published" or "2 Publications"
- "LET'S Connect!" --> "Let's Connect" (consistent capitalization, drop the exclamation mark)
- "Let's make something amazing together" --> something more specific and less overused
- "Start by saying hi" --> "Tell me about your project" or "Describe what you're looking for"
- Review all project descriptions for consistent tone

**Why it matters:** Small grammar and formatting issues undermine the professionalism of the entire site. "2 research published" is grammatically incorrect and is one of the first stats a visitor sees. Inconsistent capitalization ("LET'S Connect!") signals inattention to detail -- which is the last impression you want from a cybersecurity researcher where precision matters.

---

### 2. Reduce Font Loading (7 families to 3-4)

**What to do:** Audit which fonts are actually used and visible. Currently loading: Montserrat, Roboto, Orbitron, Share Tech Mono, Audiowide, Raleway, plus the custom CyberAlert font. Raleway appears in the CSS reset but may not be used. Consolidate to: Montserrat (headings), Roboto (body), Share Tech Mono (code/labels), and the CyberAlert font. Drop Orbitron, Audiowide, and Raleway unless they serve a clear visual purpose.

**Why it matters:** 6+ Google Font families on a single page significantly impacts load time and Core Web Vitals. 40% of recruiters check portfolios on mobile where every kilobyte matters. Fonts are render-blocking resources by default. This is a quick win that improves performance with zero visual compromise if the unused fonts are truly unused.

---

### 3. Add Open Graph and Twitter Card Meta Tags

**What to do:** Add proper `og:title`, `og:description`, `og:image`, and `twitter:card` meta tags so the portfolio looks professional when shared on LinkedIn, Twitter/X, or Slack. The og:image should be a designed card showing Mahipal's name, title, and a visual element -- not a blank preview.

**Why it matters:** When someone shares the portfolio link on social media or in a Slack channel, the preview card is often the first impression. Without OG tags, the link shows as a bare URL or pulls random page content. A well-designed social preview card turns every share into a mini-advertisement for the portfolio.

### 4. Add a "View My Work" Secondary CTA in the Hero

**What to do:** Add a ghost/outline button next to the Calendly CTA that scrolls to the projects section. Something like "See My Work" or "Explore Projects."

**Why it matters:** The current hero has only one CTA: schedule a Calendly call. That's a high-commitment action for a first-time visitor who just landed on the page. Most visitors want to browse the work first before committing to a conversation. Having a low-commitment secondary CTA ("see my work") captures the majority of visitors who aren't ready to book a call but are interested enough to keep scrolling.

## What NOT to Do

### 1. Do NOT Add a Terminal/Command-Line Interface as the Primary Navigation

While terminal-style portfolios are popular in the cybersecurity niche (and the research shows several examples), implementing one as the primary navigation would be a mistake for Mahipal's portfolio. Terminal UIs are memorable but hurt accessibility and discoverability. Many visitors (especially non-technical recruiters, academic collaborators, or potential clients like SK Studio) would find it confusing or frustrating. If a terminal mode is desired, implement it as an optional easter egg or toggle -- never as the only way to navigate the site. The portfolio's audience is broader than just hackers.

### 2. Do NOT Add 3D Animations or Heavy WebGL Effects

It's tempting to add Three.js scenes, particle effects, or 3D models to compete with portfolios like Bruno Simon's. But Bruno Simon is a creative developer whose portfolio IS his demo of 3D skills. Mahipal is a security researcher and MCP tool builder -- the portfolio should demonstrate those skills, not WebGL mastery. Heavy 3D effects would increase load times, hurt mobile performance, and send the wrong signal about professional focus. The dark theme with green accents and clean typography already establishes the right aesthetic. Invest time in better content (screenshots, case studies, metrics) rather than flashy animations.

### 3. Do NOT Add More Projects to Fill Space

The instinct to "show more work" by adding every project ever built is counterproductive. The portfolio already has projects that dilute the brand (the moving company site, the Java routing protocol assignment). Adding more mediocre projects would make this worse. The goal should be fewer, deeper, better-documented projects -- not more cards. If anything, remove 2-3 projects and invest that space into making the remaining ones extraordinary with screenshots, metrics, and narratives.
